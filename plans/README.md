This repo is a single-user personal data warehouse (Python/Dagster pipeline +
Go MCP/ingest/timeline app) that **auto-deploys to production on pushes to
`main`**. That fact drives the prioritization: cheap, high-confidence bug fixes
and a verification gate come first.

## Recommended sequencing

All five plans are **independently executable** (none edits the same lines as
another). Suggested order and why:

1. **001** and **002** first — smallest, highest-confidence, cleanest to verify;
   both are real production bugs (needless job serialization; Slack full-re-sync
   on transient errors). Neither depends on anything.
2. **003** — security hardening of the untrusted agent SQL path (read-only,
   isolated connection + a real query timeout). Self-contained code change.
3. **004** — connection-leak / churn fix across 7 per-minute sensors on the
   load-sensitive prod host.
4. **005** — the CI gate. Numbered last only because its true verification runs
   on GitHub and it likely needs operator involvement (it may surface
   pre-existing test failures, and deciding blocking-vs-non-blocking is the
   operator's call). **It is the highest-leverage systemic fix** — once it lands,
   the new tests from 001–004 (and everything else) are enforced before prod
   auto-deploys. Consider landing it early even though it's listed last, if the
   operator is available to triage any red.

## Direction — options for the maintainer (not ranked against bugs)

Grounded in repo evidence; each is a design/spike, not a build-everything plan.

- **Extend the upstream-mutation write path beyond Google (Slack first).** The
  propose→approve→execute state machine + preview UI is source-agnostic and
  already backs Gmail/Calendar/Contacts (`defs/upstream_mutations.py:102-104`),
  and a write-capable `SlackWebApiClient` already exists in `slack_sync.py`. The
  MCP server is literally named "Gmail + Slack" and exposes `propose_mutation`,
  yet Slack writes aren't executable — a visible create-without-write asymmetry.
  Trade-off: Slack sends are non-idempotent and need the same idempotency guards
  the code already flags for calendar/email.
- **Wire Google Drive "Phase 3" binary extraction to the existing
  attachment-enrichment pipeline.** Drive-sourced PDFs/Office docs are ingested
  but their text is invisible to `search_text()`
  (`google_drive_source_sync.py:105` records them `unsupported`), while
  `file_attachment_enrichment.py` + `pypdf` already extract exactly those formats
  for Gmail/Apple/WhatsApp. Trade-off: cost/rate-limit of extraction over a large
  Drive corpus — needs batch/backfill bounds like the other sources.
- **Add a semantic/vector recall layer over the unified timeline.** Search is
  BM25 + trigram only, and the MCP help text itself enumerates the recall misses
  (no phrase/typo tolerance; approximate cross-source order). An in-pipeline LLM
  enrichment container already exists, so embedding generation is in-reach; a
  `pgvector` column + hybrid rank targets the exact documented failure. Trade-off:
  embedding cost/storage, and hybrid-rank quality must be measured against the
  timeline priority benchmark to avoid regressing tuned BM25 precision.
- **Decide the fate of the stubbed S3 object-store backend — build or delete.**
  `objectstore/s3.py` is a documented "not implemented yet" seam alongside the
  live Google-Drive backend; the Drive coupling carries a real constraint (the
  Cloudflare 100 MiB upload-cap workaround). Either implement it (MinIO on the
  existing host) to decouple, or delete the stub as dead code. Low-confidence on
  which — it's a maintainer call about whether Drive-coupling is worth removing.

## Findings considered and rejected

Recorded so they aren't re-audited next run:

- **Go MCP/ingest server per-request client construction / unbounded reads** —
  not present. DB pools and Drive stores are built once at startup and injected
  (`app/cmd/pdw-mcp/main.go:28`, `server.go:242,290`, `objects_http.go:25`);
  uploads are size-capped (`ingest.go:482`).
- **`timeline.py` full-rebuild on every sync** — not a bug; it's watermark-driven
  incremental with a bounded refresh window (`timeline.py:1583-1699`), inserts
  batched via `execute_values`. The per-refresh re-walk is a deliberate, bounded
  tradeoff.
- **Core Postgres insert path row-at-a-time** — already batched via
  `execute_values` with per-table page sizes (`postgres.py:6664-6687`).
- **HMAC upload/download replay / body-tamper / non-constant-time compare** — the
  scheme is sound: signatures bind endpoint + body sha256, enforce expiry, and use
  constant-time comparison (`server/ingest.go:159-166`, `auth/*.go`). (The MAC
  field-encoding nit #21 is a separate, latent, low-severity item.)
- **Mutation-console CSRF / session-cookie hardening** — already enforced:
  per-form CSRF on every state-changing POST; HMAC-signed, HttpOnly, SameSite=Lax,
  Secure-when-https session cookie (`mutations/ui.go:1747-1792`).
- **Object proxy SSRF/IDOR** — not a sink; serves only signature-verified,
  server-minted file ids with the account bound into the signature
  (`server/objects_http.go:31-53`).
- **Python SQL injection in query building** — user values are parameterized;
  `.format()`/f-string SQL is limited to static fragments or escaped
  information_schema identifiers. (The agent-proxy issue #3/#4 is about the
  *connection privilege/timeout*, not injection.)
- **`neonize` major-version bump** — deliberately pinned `0.3.17.post0` (dagster
  protobuf conflict), documented; not a finding.
- **Go/Python dependency major-version lag** — none with real cost; deps are
  current (Go shows only minor/patch updates; Python core libs current). No
  duplicate-purpose deps.
